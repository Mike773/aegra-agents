# Langgraph executor


## Общее описание

> Мы выводим на UAT **Aegra Agents** — набор ИИ-ассистентов для руководителей,
> которые помогают разбираться в показателях сотрудников в формате диалога. Руководитель может спросить про метрики подчинённого,
> получить разбор отклонений от плана, трендов и аномалий.

## Что такое Aegra

**Aegra** — это open-source платформа для запуска и обслуживания ИИ-агентов,
self-hosted альтернатива облачному LangGraph Platform. Она разворачивается на
нашей инфраструктуре (никакие данные не уходят во внешние сервисы) и
предоставляет стандартный HTTP API для работы с агентами:

- `/assistants` — реестр доступных агентов;
- `/threads` — диалоговые сессии (с сохранением состояния);
- `/runs/stream` — запуск агента с потоковой (streaming) отдачей ответа.

В нашем проекте Aegra выступает «сервером», на котором зарегистрированы
агенты. LLM и эмбеддинги для них работают через
**GigaChat** (`langchain-gigachat`), модель `GigaChat-2-Max`.


## Что такое LangGraph

**LangGraph** — это фреймворк для построения ИИ-агентов в виде **графа
состояний (state machine)**. Вместо одного запроса к модели логика агента
описывается как набор шагов («узлов») и переходов между ними:

- **Узлы (nodes)** — отдельные шаги: загрузить данные, обратиться к LLM,
  сходить в базу, сформировать ответ.
- **Состояние (state)** — структура данных, которая передаётся между узлами и
  накапливает контекст диалога.
- **Маршрутизация (conditional edges)** — переходы зависят от условий:
  например, по «намерению» пользователя запрос уходит либо в анализ метрик,
  либо в поиск по вики, либо в обычный ответ.

Что это даёт нашему проекту:

- **Многошаговые сценарии** вместо «вопрос → ответ»: загрузка метрик → анализ →
  уточнение у пользователя → вызов нужного под-агента → ответ.
- **Human-in-the-Loop**: агент может приостановиться (`interrupt`), задать
  уточняющий вопрос и продолжить с того же места после ответа пользователя.
- **Сохранение состояния диалога** между репликами (checkpointing)
- **Композиция агентов**: крупный оркестратор переиспользует более простые
  графы как подграфы.
- **Потоковая выдача (streaming)** ответов клиенту по мере выполнения шагов. (Пока у нас не используется, мы в синхронном режиме работаем)

Проще говоря: Aegra — это «сервер», на котором живут агенты, а LangGraph —
«язык», на котором описана внутренняя логика каждого агента.

## Как используется SQLite

**SQLite в проекте используется не как постоянная база
данных, а как временное аналитическое хранилище в оперативной памяти.**

Когда пользователь спрашивает про метрики сотрудника, агент `json_analyzer`:

1. Создаёт **SQLite-базу в памяти** (`:memory:`) — свежую на каждый запрос,
   без файлов на диске. **По времени все занимает 100-200 миллисекунд** 
2. Загружает в неё присланный JSON с метриками в таблицу `metrics`.
3. Считает производную аналитику и складывает её в таблицу `metric_analytics`:
   отклонения от плана и бенчмарка, динамику неделя-к-неделе, тренды, сравнение
   с коллегами (peer-ранг, перцентиль), z-score и признак аномалии.
4. По этой in-memory базе LLM-агент с помощью инструментов (tool-calling)
   собирает факты и формирует ответ на вопрос пользователя.

Почему именно так:

- **Скорость и удобство аналитики** — SQL-агрегаты по метрикам считать проще и
  быстрее, чем перебирать JSON вручную.
- **Безопасность** — llm не генерит sql, а вызывает предварительно разработанные tools, в которых описаны все запросы. Т.е. gigachat выбирает нужные инструменты и подставляет параметры для их запуска. 

### Чем отличается от PostgreSQL в проекте

Для **постоянного** хранения используется не SQLite, а
**PostgreSQL + pgvector**:

- схема `json_analyzer` — кэш эмбеддингов названий/описаний метрик для «умного» сопоставления нечётко заданных названий метрик;

Хранилища разделены по `direction_key` (направление/подразделение), что
обеспечивает изоляцию данных между подразделениями.
