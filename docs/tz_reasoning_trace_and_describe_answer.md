# ТЗ: сквозной лог шагов рассуждения, `describe_answer` и углубление анализа

**Статус:** черновик к согласованию
**Дата:** 2026-05-31
**Связанные документы:** [`ideal_answers.md`](../ideal_answers.md) — эталон глубины и формата ответа аналитика.

---

## 1. Контекст и цель

Бизнес ожидает от агента два качественных скачка:

1. **Глубина анализа.** Ответы должны соответствовать методологии из `ideal_answers.md`:
   многоуровневый анализ (опережающие/запаздывающие метрики, производные показатели,
   сегментация, контрфактуалы, отбраковка ложных корреляций), приоритизированная карта
   инсайтов и связный объясняющий нарратив, а не констатация «метрика упала/выросла».

2. **Объяснимость (explainability).** Агент должен на каждом запросе формировать
   **внутренний лог шагов** («из базы знаний нашёл описание X → в метриках нашёл Y →
   подтянул связанные метрики Z → проверил гипотезу → вывод»). Если в запросе передан
   параметр `describe_answer=true`, агент дополнительно выводит пользователю
   **человекочитаемый раздел «Как я пришёл к выводу»** с разворачиванием этого лога.

Эти два требования связаны: лог шагов — это материал для разделов «Доказательная база»
и «Обоснование выбора приоритетов» из формата ответа в `ideal_answers.md`.

### Решения по объёму (согласовано)

- **Охват лога:** весь конвейер оркестратора — роутинг (выбранный intent), находки
  `easyrag` (какие секции wiki, similarity), tool-вызовы `json_analyzer` (какие метрики,
  связанные метрики), извлечение/фиксация поручений.
- **Формат выдачи `describe_answer`:** дополнительный человекочитаемый раздел в тексте
  того же ответа (фронту не нужно парсить новые поля).
- **Методология `ideal_answers.md`:** входит в это ТЗ (апгрейд синтез-промпта и формата
  ответа `json_analyzer` и респондера оркестратора).

## 2. Текущее состояние (как есть)

### 2.1. Оркестратор (`analytic_orchestrator`)

Turn-based граф (`graph.py`): один вызов = одно входящее сообщение → ответ → `END`,
состояние держится per-thread чекпоинтером aegra.

- `load_data` → грузит датасет метрик по оргструктуре.
- `initial_analysis` → первичный обзор LLM по **обрезанному** JSON (`_METRICS_PREVIEW_LIMIT = 8000`).
- `extract_assignments` → гоняет `json_analyzer` фиксированным вопросом про проблемные зоны,
  затем вторым LLM-вызовом структурирует в кандидатов-поручения.
- `route` → классифицирует реплику в `intent` (`analytics | wiki | chat | done |
  assignments | assignments_select`).
- `call_json_analyzer` / `call_easyrag` → вызывают соответствующие подграфы.
- `respond` → финальный ответ: системный промпт + блок метрик (`_metrics_system_block`) +
  блок wiki (`_easyrag_system_block`) + история сообщений.
- ветка поручений: `propose_assignments` → `select_assignments` → `commit_assignments`.

**Что уже есть для лога, но выбрасывается:**
- `easyrag` возвращает `snippets` — секции wiki с `page_title`, `section_title`,
  `similarity`, `body_md`. Это готовый «что нашли в базе знаний».
- `analytics_answer` сохраняется, но **трасса рассуждения — нет**.

### 2.2. json_analyzer

Двухстадийный подграф (`graph.py`): `gather` → `synthesize`.

- `gather` (стадия 1): tool-loop агент (`agent_classic.ClassicStrategy`) по in-memory
  SQLite + индексу эмбеддингов. Инструменты (`tools.py`): `schema_overview`,
  `resolve_entity`, `describe_metric`, `get_metric`, `compare`, `rank`, `aggregate`,
  `metric_tree`, `list_people`, `find_flags`, `analytics_summary`.
  Транскрипт «вызов инструмента → результат» собирается в `gathered_facts` через
  `agent_base.extract_tool_transcript()` (формат: `N. tool(args) ->\n результат`).
- `synthesize` (стадия 2): один LLM-вызов без инструментов по `SYNTHESIS_PROMPT`,
  превращает транскрипт в финальный `answer`.

**Ключевой факт:** `gathered_facts` — это уже почти готовый лог шагов анализа метрик.
Он формируется на каждом вызове, но потребляется только синтезом и наружу не отдаётся.

## 3. Объём доработок

Три блока, реализуются последовательно (A — фундамент, B — поверх него, C — параллельно).

### Блок A. Сквозной лог шагов (reasoning trace)

Агент на каждом запросе накапливает структурированный список шагов. Это **внутренняя**
структура (формируется всегда, независимо от `describe_answer`).

#### A.1. Модель данных шага

Единый формат записи шага (далее — `TraceStep`):

```python
class TraceStep(TypedDict, total=False):
    stage: str        # узел конвейера: "route" | "easyrag" | "json_analyzer"
                      #                 | "assignments" | "respond"
    kind: str         # тип шага: "intent" | "kb_hit" | "tool_call"
                      #          | "derived_metric" | "hypothesis" | "decision"
    summary: str      # человекочитаемая строка одной фразой
    detail: dict      # машиночитаемые детали (имя инструмента, аргументы,
                      # similarity, имена метрик, числа и т.п.)
```

#### A.2. Хранение трассы в состоянии

В `OrchestratorState` (`analytic_orchestrator/state.py`) добавить поле:

```python
reasoning_trace: list[TraceStep]
```

**Жизненный цикл — строго per-turn** (трасса описывает рассуждение текущего хода, не
копится между ходами как `messages`):

- Точка сброса — первый узел хода: `route` (на последующих ходах) и `load_data`
  (на первом ходе) перезаписывают `reasoning_trace` свежим списком.
- Downstream-узлы дополняют трассу, **читая текущее значение и возвращая
  `state.get("reasoning_trace", []) + new_steps`** (плоское поле TypedDict
  перезаписывается, поэтому накопление — явным конкатенированием в каждом узле;
  reducer `operator.add` НЕ использовать — он бы копил трассу и между ходами).

#### A.3. Что писать в каждом узле

| Узел | `stage` | Что логировать |
|------|---------|----------------|
| `route` | `route` | `kind="intent"`: выбранный intent + краткое «почему» (по тексту реплики). Сброс трассы. |
| `call_easyrag` | `easyrag` | по одному `kind="kb_hit"` на сниппет: `page_title / section_title`, `similarity`, краткая выжимка `body_md`. Если пусто/ошибка — шаг с пометкой «в базе знаний релевантного не нашлось» / причиной ошибки. |
| `call_json_analyzer` | `json_analyzer` | развернуть `gathered_facts` подграфа в шаги `kind="tool_call"` (имя инструмента, аргументы, краткая сводка результата). См. A.4. |
| `extract_assignments` | `assignments` | шаги по анализу проблемных зон (из `json_analyzer`) + итог: сколько кандидатов извлечено. |
| `select_assignments` / `commit_assignments` | `assignments` | какой выбор распознан, что ушло в mock-сервис / ошибка. |
| `respond` | `respond` | финальный `kind="decision"`: на какие блоки (метрики/wiki) опирался ответ. |

#### A.4. Проброс трассы из подграфа json_analyzer наверх

`json_analyzer` должен вернуть структурированные шаги, а не только текстовый
`gathered_facts`.

- В `JsonAnalyzerState` (`json_analyzer/state.py`) добавить поле `tool_steps: list[dict]`.
- Доработать `agent_base.extract_tool_transcript()` (или добавить рядом
  `extract_tool_steps()`), чтобы помимо строки-транскрипта возвращать список структур:
  `{"tool": name, "args": {...}, "result_summary": "<сжатая выжимка>"}`.
  Источник тот же — `ToolMessage` + `tool_calls`, уже разбираемые в этой функции.
- `gather`-узел кладёт `tool_steps` в стейт; `call_json_analyzer` в оркестраторе
  читает их из результата `ainvoke` и мапит в `TraceStep` со `stage="json_analyzer"`.

**Изоляция ошибок (как сейчас):** если подграф упал, трасса получает один шаг с
причиной ошибки, конвейер не падает.

### Блок B. Параметр `describe_answer` и раздел «Как я пришёл к выводу»

#### B.1. Проброс параметра

`describe_answer` приходит через `config["configurable"]` — там же, где `boss_tabnum`,
`employee_tabnum`, `easyrag_enabled`, `easyrag_top_k`, `system_prompt_override`
(см. `make_load_data_node`, `make_respond_node`).

- Тип: `bool`, дефолт `false`.
- Нормализация: принимать `true/false`, строки `"true"/"1"/"yes"` — на случай прокидывания
  из внешнего API строкой.

#### B.2. Поведение

- `describe_answer=false` (дефолт): поведение как сейчас — трасса копится внутри, но в
  ответ **не** добавляется. Текст ответа не меняется (обратная совместимость).
- `describe_answer=true`: к финальному ответу `respond` (и к ответам ветки поручений, где
  это осмысленно) добавляется отдельный раздел:

  ```
  ---
  ### Как я пришёл к выводу

  1. **Классификация запроса.** Определил тип вопроса как «аналитика по метрикам».
  2. **База знаний (wiki).** Нашёл описание метрики «…» (раздел «…», релевантность 0.82):
     «…».
  3. **Метрики.** Через `find_flags(kind="below_plan")` нашёл просадку по «…» (факт … /
     план …). Подтянул связанные: `metric_tree("…")` показал, что просадку тянет
     компонент «…».
  4. **Проверка.** Сопоставил с трендом (`compare`) — спад устойчив 3 недели.
  5. **Вывод.** …
  ```

#### B.3. Способ генерации раздела

Раздел строится **из `reasoning_trace`**, накопленной в блоке A. Два варианта реализации
(выбрать на этапе проектирования, см. открытые вопросы):

- **B.3.a. Детерминированный рендер** — функция `render_trace(trace) -> str` собирает
  markdown-список напрямую из `TraceStep`. Плюс: дёшево, точно, без галлюцинаций. Минус:
  суше по языку.
- **B.3.b. LLM-перефраз** — отдельный LLM-вызов с трассой на входе и инструкцией изложить
  ход рассуждения связным текстом (стиль «рассказываем историю» из `ideal_answers.md`,
  этап 6). Плюс: читабельнее. Минус: токены + риск, что модель додумает шаги, которых не
  было. **Митигировать** жёстким системным промптом «опирайся ТОЛЬКО на переданные шаги,
  ничего не добавляй».

**Рекомендация:** B.3.a как база (всегда корректно), с опциональным B.3.b как
надстройкой позже.

#### B.4. Где добавляется раздел

- В `respond` (`make_respond_node`): после получения `ai`, если `describe_answer=true`,
  дописать раздел к `ai.content`.
- В ветке поручений (`commit_assignments`, `propose_assignments`): по решению — для
  поручений объяснение «почему именно эти зоны» особенно ценно бизнесу.

### Блок C. Углубление анализа под методологию `ideal_answers.md`

Цель — приблизить выходные ответы к эталону: многоуровневый анализ + приоритизированная
карта инсайтов + объясняющий нарратив.

#### C.1. Синтез-промпт json_analyzer (`json_analyzer/prompts.py` → `SYNTHESIS_PROMPT`)

Переписать под методологию `ideal_answers.md`:
- роль «ведущий аналитик-исследователь», поиск неочевидных взаимосвязей;
- требование строить цепочку «входные действия → промежуточный → конечный результат»,
  различать опережающие и запаздывающие метрики;
- считать производные показатели («цена результата», истинная конверсия, КПД), когда
  данные позволяют;
- формат ответа: Executive Summary → приоритизированная карта инсайтов (формулировка →
  доказательная база → корневая причина → бизнес-влияние) → обоснование приоритетов →
  неочевидные находки → гипотезы для проверки;
- ограничение «не более 3–5 инсайтов» (принцип Парето);
- **жёсткое** «не выдумывай данных, которых нет в собранных фактах».

#### C.2. Инструменты для взаимосвязей и производных метрик

> Уточнено по итогам ревью `analytics.py` и `sqlite_store.py` (2026-05-31).

**Что методология уже закрывает существующей детерминированной аналитикой —
новых инструментов НЕ требует:**

| Техника `ideal_answers.md` | Чем закрыта |
|----------------------------|-------------|
| Отклонения от плана/нормы, масштаб отклонения (этапы 2, 3) | `plan_status`, `plan_dev_pct/abs`, `benchmark_*` (учитывают направление метрики `прямая`/`обратная`) → `find_flags(below_plan/above_plan)`, `get_metric` |
| Тренд / динамика, отложенные эффекты — материал (этап 4.А) | `pop_change_pct/abs`, `trend` (рост/падение/стабильно) → `compare`, `find_flags(trend)` |
| Внутренние контрасты между людьми (этап 4.В) | `peer_rank`, `peer_percentile`, `zscore`, `peer_mean/std` → `rank`, `find_flags(anomaly)` |
| Сегментация по дням/разрезам/должностям (этап 4.В) | `aggregate(group_by=person\|element\|date\|post)` + фильтры `element/date/post` |
| Декомпозиция «что тянет результат» (этап 4, корневые причины) | `metric_tree` (иерархия parent/child + `influent_percent` + аналитика по каждому узлу) |
| Выбросы / аномалии (этап 2) | `is_anomaly` (порог z-score, env `ANOMALY_ZSCORE_THRESHOLD`) |
| Стартовая сводка для осмотра (этап 2) | `analytics_summary` (охват, средние на последнем периоде, топ-аномалии, счётчики трендов) |
| Приоритизация инсайтов (этап 5) | Не инструмент, а синтез: данные для матрицы есть (`plan_dev_pct` = масштаб, `trend` = срочность, `peer_rank` = относительная позиция) |
| Контрфактуалы (этап 4.Д), отбраковка ложных корреляций (этап 4.Е) | Рассуждение LLM по уже доступным числам + бизнес-контекст из wiki; отдельный инструмент не нужен |

**Реальные дыры (по приоритету; реализовывать только при подтверждённой потребности):**

1. **Опережающие/запаздывающие метрики не размечены** (этапы 1, 4.А) — *приоритет: высокий.*
   В данных есть только иерархия `parent_uid/depth` (структурная декомпозиция), а не
   причинно-следственная цепочка «вход → результат». **Основное решение — блок D**
   (LLM-граф смысловых связей из названий/описаний: он размечает связи вида
   «опережающая→запаздывающая» и «влияет_на»). Дополнительно знание о цепочке ценности
   живёт в **wiki** (см. C.4). Сопоставить две серии по лагу агент уже может двумя вызовами
   `compare()` + рассуждение.
   *Опциональный инструмент:* `compare_metrics(metric_a, metric_b, person)` — выровненные
   по периодам серии двух метрик рядом, чтобы рассуждение про лаг было надёжным, а не
   «на глаз» по двум отдельным выдачам.

2. **Нет кросс-метричной арифметики** (этап 4.Б: «цена результата», КПД, истинная
   конверсия) — *приоритет: средний, условный.*
   `aggregate` считает avg/min/max/sum/count по **одной** метрике; отношения метрики A к
   метрике B (часы/результат и т.п.) не поддерживаются.
   **Сначала проверить на реальных данных**, не заведены ли такие коэффициенты уже
   отдельными метриками в датасете (тогда инструмент не нужен). Если нет — добавить
   `ratio(numerator_metric, denominator_metric, person, date)` либо считать производные в
   `analytics.compute_analytics`.

3. **Дисперсия только межчеловеческая** (этап 4.Г «тихие убийцы» — стабильное среднее при
   огромном разбросе) — *приоритет: низкий.*
   `peer_std`/`zscore` — разброс между людьми на одну дату. Разброс одной метрики во
   времени или по разрезам `element` одного человека не выводится. При необходимости —
   добавить показатель дисперсии в `compute_analytics`.

> Не плодить инструменты заранее: пункты 1–3 — кандидаты, а не обязательный объём.
> Базовый эффект методологии даёт C.1 (синтез-промпт) + C.3 (промпт стадии gather) + C.4.

#### C.3. Промпт стадии gather (`agent_classic` / `SYSTEM_PROMPT_RULES`)

Методологию реализует не только синтез: сбор фактов (стадия 1) должен **активно**
исследовать связи, а не отвечать первым же `get_metric`. Дополнить системный промпт
стадии gather инструкциями: при просадке — разложить метрику через `metric_tree`,
сверить с трендом (`compare`) и с планом (`find_flags`), сопоставить с peer-позицией
(`rank`). Учитывать бюджет инструментов (`_TOOL_CALL_BUDGET = 18`) — не раздувать.

> Нюанс по `find_flags(kind='trend')`: сортировка идёт по `pop_change_pct` (изменение
> последнего периода), а не по совокупному тренду серии. Для «самого сильного спада за
> период» это приближение — учесть в промпте или скорректировать сортировку.

#### C.4. Бизнес-контекст из wiki в аналитику (архитектурная доработка)

**Находка:** `json_analyzer` сейчас НЕ видит wiki — он получает только метрики. В
оркестраторе ветки `analytics` и `wiki` взаимоисключающи за один ход (`after_route`), а
`call_json_analyzer` не передаёт в подграф контекст easyrag. Но этап 1 методологии
(«Погружение в бизнес-контекст», построение модели «входные действия → промежуточный →
конечный результат», различение опережающих/запаздывающих метрик) опирается именно на
базу знаний. Сейчас глубокий аналитический ответ синтезируется **без модели цепочки
создания ценности**.

**Варианты (выбрать на проектировании):**
- **C.4.a.** Перед/во время `call_json_analyzer` подтянуть релевантные wiki-секции
  (вызвать easyrag по вопросу) и прокинуть их в синтез json_analyzer как бизнес-контекст.
- **C.4.b.** Дать стадии синтеза доступ к wiki-блоку, который уже умеет собирать
  оркестратор (`_easyrag_system_block`), для аналитических ходов тоже.

Это дополняет дыру №1 (lead/lag) бизнес-контекстом из wiki — поверх основного механизма
(LLM-граф связей, блок D). Объём уточнить — возможно вынести в отдельную итерацию, если
ломает текущий turn-based поток.

#### C.5. Согласованность с трассой

Поскольку синтез теперь даёт структурированные разделы (инсайты с доказательной базой),
раздел «Как я пришёл к выводу» (блок B) и «Доказательная база» инсайта должны ссылаться
на одни и те же шаги трассы — без расхождений.

---

### Блок D. LLM-граф смысловых связей между метриками (замена/дополнение `influent_percent`)

#### D.1. Проблема

`influent_percent` (вес влияния дочерней метрики на родителя) проставляет бизнес вручную и
**часто забывает/ошибается** → поле `null` или неверно. Сейчас оно используется только как
презентационное в выводе `metric_tree` (в расчётах не участвует): LLM смотрит на него,
чтобы понять, какой компонент тянет просадку родителя. Когда пусто — сигнал теряется.
Кроме того, иерархия `parent_uid/depth` отражает структурную декомпозицию, но не
причинно-следственные и кросс-деревьевые связи между метриками (см. C.2, дыра №1).

#### D.2. Решение (согласовано)

Достроить **дополнительный граф смысловых связей**, выведенный **LLM** из **названий и
описаний** метрик (явное решение: **не эмбеддинги** — косинус даёт лишь поверхностную
близость; LLM выводит причинно-следственные и lead/lag связи «звонки → разговоры → лиды →
продажи»). Эмбеддинги остаются только под `resolve_entity`.

- **Охват:** любая метрика ↔ любая (не только siblings внутри родителя) — ловим
  кросс-деревьевые связи, которые иерархия и `influent_percent` пропускают.
- **Выдача:** новый инструмент `related_metrics(metric)` + обогащение `metric_tree` там,
  где `influent_percent` пуст.
- **Природа:** эвристика, не факт. Связи выведены из текста, не из значений — в выводах
  агента и в ответе подаются как «предположительно влияет на…», никогда как точный вес.

Граф зависит только от каталога метрик направления (имена/описания/тип/родитель), не от
значений `fact` → стабилен per-direction и **кэшируется** в LangGraph Store (LLM дёргается
лишь на холодном кэше). Архитектура повторяет существующий путь эмбеддингов
(`store_cache.sync_embeddings` / namespace `metric_embeddings` / таблица `metric_analytics`).

#### D.3. Состав доработок

- **Новый модуль `json_analyzer/relations_cache.py`** (аналог `store_cache.py`): построение
  каталога метрик, хэширование каталога (`catalog_hash` = ключ инвалидации кэша),
  кэш-I/O в Store (namespace `("json_analyzer", "metric_relations", direction_key)`),
  LLM-вызов **без инструментов** (как `synthesize` — лимит 4096 токенов не действует) в
  `asyncio.to_thread`, строгий парсинг/валидация JSON, чанкинг больших каталогов.
  Капы: `_MAX_CATALOG_METRICS=60`, `_MAX_EDGES=120`, `_CHUNK_METRICS=40`.
- **`json_analyzer/nodes.py` (`gather`):** после `sync_embeddings` вызвать
  `sync_relations(...)` и `await asyncio.to_thread(store.load_relations, relations)`.
  `state.py` не меняется (связи живут в SQLite + кэш Store).
- **`json_analyzer/sqlite_store.py`:** новая таблица `metric_relations`
  `(source, target, relation, strength, rationale)` + индексы; методы `relation_catalog()`,
  `load_relations(edges)`, `related_metrics(name)`; `_annotate_tree_relations(rows)` —
  в `metric_tree` для узлов с `influent_percent IS NULL` подмешивает
  `influent_percent_missing: True` и `inferred_relation` (relation/strength/rationale) из
  графа по паре ребёнок↔родитель; `inferred_relation: None`, если ребра нет (граф спрошен —
  пусто, чтобы не путать с «графа нет»).
- **`json_analyzer/tools.py`:** новый инструмент `related_metrics(metric)` (валидация через
  `_unknown_metric`, компактный JSON), регистрация в `specs`. Docstring подчёркивает
  эвристичность и пользу при пустом `influent_percent`.
- **`json_analyzer/prompts.py`:** новый `RELATIONS_SYSTEM_PROMPT` (строгий JSON-массив рёбер
  `{source, target, relation∈{опережающая→запаздывающая,компонент,смежная,влияет_на},
  strength∈{низкая,средняя,высокая}, rationale}`, только осмысленные связи, имена дословно);
  правка `SYSTEM_PROMPT_RULES` — когда звать `related_metrics` и как читать аннотации
  `metric_tree`; (опц.) правка `SYNTHESIS_PROMPT` — подавать связи как предположительные.

#### D.4. Изоляция ошибок

LLM/парсинг падает → `_infer_edges` возвращает `None` → в SQLite грузится `[]`, **кэш НЕ
пишется** (чтобы транзиентный сбой не отравил кэш; легитимный пустой `[]` от LLM —
кэшируется). Сбой Store-операций — в `try/except`, без проброса. `gather` не ломается.

#### D.5. Критерии приёмки (дополнительно к §6)

1. `related_metrics("<метрика>")` возвращает рёбра, где метрика участвует как source или
   target, сильные связи первыми.
2. `metric_tree` для дочернего узла с `influent_percent = null` отдаёт
   `influent_percent_missing: true` и `inferred_relation` (или `null`, если связи нет).
3. Повторный запрос по тому же направлению не вызывает LLM (граф из кэша Store); изменение
   набора/описаний метрик инвалидирует кэш и пересчитывает граф.
4. Связи везде помечены как предположительные/эвристические, не как точный вес.
5. Сбой LLM-вывода связей не валит `gather`: `related_metrics` отвечает «связей нет»,
   аннотации `metric_tree` отсутствуют.

#### D.6. Риски

Инвалидация кэша — через `catalog_hash` (имена+описания+тип+ед.+родитель+has_influent);
пустые описания — метрика остаётся (имя+тип несут сигнал); большой каталог — кап + чанкинг
(за капом отбрасываются низкоприоритетные метрики — логировать); O(N²) рёбер — кап
`_MAX_EDGES` + инструкция в промпте; лишний LLM-вызов только на холодном кэше per-direction.

> Подробная пошаговая реализация — в plan-файле `async-petting-cloud.md`.

## 5. Нефункциональные требования

- **Обратная совместимость.** При `describe_answer=false` (дефолт) внешнее поведение и
  текст ответов не меняются. Существующие интеграции не ломаются.
- **Бюджет токенов.** Трасса не раздувает контекст синтеза: в `detail` шагов кладутся
  сжатые выжимки, не сырые tool-выдачи. Учитывать лимит GigaChat ~4096 токенов на запрос
  с функциями (см. `agent_classic`) — трасса формируется ВНЕ tool-loop.
- **Изоляция ошибок.** Падение любого подграфа/узла фиксируется в трассе отдельным шагом,
  но не валит конвейер (сохранить текущий стиль `try/except` с записью причины).
- **Конкурентность.** Счётчики и трасса — в стейте графа / локальном run-state, не в
  модульных переменных (как уже сделано в `agent_classic._RunState`).
- **Язык.** Все пользовательские тексты — на русском.
- **Приватность.** Раздел «Как я пришёл к выводу» не должен раскрывать табельные номера/PII
  сверх того, что уже есть в обычном ответе.

---

## 6. Критерии приёмки

1. На любой аналитический запрос в состоянии графа после хода присутствует непустая
   `reasoning_trace`, покрывающая роутинг + использованные источники (wiki и/или метрики).
2. При `describe_answer=true` ответ содержит раздел «Как я пришёл к выводу», в котором:
   - перечислены реально использованные источники из wiki (с указанием раздела/релевантности);
   - перечислены реально вызванные инструменты по метрикам с ключевыми находками;
   - показаны связанные метрики (если подтягивались);
   - нет шагов/фактов, которых не было в трассе (проверка на отсутствие галлюцинаций).
3. При `describe_answer=false` текст ответа идентичен поведению до доработки.
4. Ответ на содержательный аналитический вопрос соответствует структуре из
   `ideal_answers.md` (Executive Summary + карта инсайтов + обоснование), не более 3–5
   инсайтов.
5. Ошибка подграфа (easyrag/json_analyzer) → в трассе шаг с причиной, конвейер отвечает
   деградированно, не падает.
6. Существующие тесты проходят; добавлены тесты на: формирование трассы, рендер раздела,
   поведение при `describe_answer=false/true`, изоляцию ошибок.

## 7. Этапность

1. **A — фундамент.** `TraceStep`, поле в стейте, запись шагов во всех узлах, проброс
   `tool_steps` из json_analyzer. (Внутреннее, без изменения ответов.)
2. **B — explainability.** `describe_answer` из configurable, `render_trace`, раздел в
   `respond`. Тесты на обратную совместимость.
3. **C — методология.** Переписать `SYNTHESIS_PROMPT` и промпты респондера; по итогам
   ревью `analytics.py`/`tools.py` — точечные доработки инструментов.
4. **D — граф связей.** `relations_cache.py`, таблица `metric_relations`, инструмент
   `related_metrics`, обогащение `metric_tree`, промпты. Самостоятельный блок — можно
   делать параллельно A–C; усиливает C (методология lead/lag) и блок A (шаг «связанные
   метрики» в трассе).

## 8. Открытые вопросы

1. **Рендер раздела:** детерминированный (B.3.a) или LLM-перефраз (B.3.b) на старте?
   Рекомендация — начать с детерминированного.
2. **Объём блока C.2 (уточнено):** базовая аналитика уже покрывает отклонения, тренды,
   peer-контрасты, сегментацию и декомпозицию (см. таблицу в C.2). Подтвердить у бизнеса,
   нужны ли сверх этого: (а) выровненное сопоставление двух метрик по лагу
   (`compare_metrics`); (б) кросс-метричные коэффициенты «цена результата/КПД» (`ratio`) —
   *сперва проверить, нет ли их уже отдельными метриками в реальном датасете*;
   (в) внутривременная/внутрисегментная дисперсия. Без подтверждения — не реализуем.
3. **Wiki в аналитику (C.4):** включать бизнес-контекст из базы знаний в синтез
   аналитических ответов в этой итерации или вынести отдельно? Без этого этап 1
   методологии (модель цепочки ценности, lead/lag) реализуется лишь частично.
4. **Поручения и `describe_answer`:** добавлять ли раздел-объяснение в ветке поручений
   (почему именно эти зоны) или только в аналитических ответах?
5. **Глубина трассы по wiki:** логировать все найденные сниппеты или только реально
   попавшие в контекст ответа (top-K по similarity)?
6. **Канал параметра:** `describe_answer` точно приходит в `configurable`, или нужно также
   уметь читать его из тела сообщения/стейта?
